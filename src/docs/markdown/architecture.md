架构
====

Caddy 是一个单一的、自包含的、静态二进制文件，没有任何外部依赖，因为它是用 Go 编写的。这些值构成了项目愿景的重要组成部分，因为它们简化了部署并减少了生产环境中的麻烦排查工作。

如果没有动态链接，那又如何扩展呢？Caddy 拥有新颖的插件架构，其功能扩展远超任何其他 Web 服务器，即使那些具有外部（动态链接）依赖的服务器也无法相比。

我们的"更少组件"理念最终带来更可靠、更易管理、更经济的站点——尤其是在大规模部署时。这份半技术性文档描述了我们要如何通过软件工程实现这一目标。


## 概述

Caddy 由一个命令、核心库和模块组成。

**命令** 提供 [命令行接口](/docs/command-line)（希望您已经熟悉）。这是您从操作系统启动进程的方式。这里的代码和逻辑量相当少，仅包含以用户期望方式引导核心所需的内容。我们有意避免使用标志和环境变量进行配置，除了那些与引导配置相关的内容。


<aside class="tip">

模块可以向命令行接口添加子命令！例如，[`caddy file-server`](/docs/command-line#caddy-file-server) 命令就是这么来的。这些添加的命令可以有任何想要的标志或使用任何环境变量，尽管核心 Caddy 命令尽量减少使用它们。

</aside>


**[核心库](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc)**，或 Caddy 的"核心"，主要管理配置。它可以 [`Run()`](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc#Run) 新配置或 [`Stop()`](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc#Stop) 运行中的配置。它还为用户提供各种工具、类型和值供模块使用。

**模块** 做其他所有事情。许多模块内置于 Caddy 中，这些被称为_标准模块_。这些被认为是大多数用户最有用的。


<aside class="tip">

有时术语 _module_（模块）、_plugin_（插件）和_extension（扩展）会互换使用，通常这没问题。严格来说，所有模块都是插件，但并非所有插件都是模块。模块是特定类型的插件，用于扩展 Caddy 的 [配置结构](/docs/json/)。

</aside>




## Caddy 核心

在其核心，Caddy 仅加载初始配置（"config"），或者如果没有配置，则打开一个套接句以稍后接受新配置。

[Caddy 配置](/docs/json/) 是一个 JSON 文档，在其顶层有一些字段：

```json
{
	"admin": {},
	"logging": {},
	"apps": {•••},
	...
}
```

Caddy 核心原生知道如何处理其中一些字段：

- [`admin`](/docs/json/admin/) 以便它可以设置 [管理 API](/docs/api) 并管理进程
- [`logging`](/docs/json/logging/) 以便它可以 [发出日志](/docs/logging)

但其他顶级字段（如 [`apps`](/docs/json/apps/)）对 Caddy 核心而言是透明的。事实上，Caddy 对 `apps` 中的字节唯一知道的操作是将它们反序列化为接口类型，该接口可以调用两个方法：

1. `Start()`
2. `Stop()`

...仅此而已。Caddy 在加载配置时会对每个应用调用 `Start()`，在卸载配置时对所有应用调用 `Stop()`。

当应用模块启动时，它会启动应用模块的生命周期。


<aside class="tip">

如果您是构建 Caddy 模块的程序员，您可以在 [扩展 Caddy](/docs/extending-caddy) 指南中找到类似的信息，但更侧重于代码。

</aside>


## 模块生命周期

有两种模块：_宿主模块_和_访客模块_。

**宿主模块**（或"父"模块）是加载其他模块的模块。

**访客模块**（或"子"模块）是被加载的模块。所有模块都是访客模块——即使应用模块也是如此。

模块按以下顺序被加载、准备和验证、使用，然后清理：

1. 加载
2. 准备和验证
3. 使用
4. 清理

Caddy 在加载配置时首先通过初始化所有配置的应用模块来启动模块生命周期。从此，每一步都是层层深入，每个应用模块带领完成剩余部分。

### 加载阶段

加载模块涉及将其 JSON 字节反序列化为内存中的Typed 值。那就是...基本上就是这样。只是将 JSON 解码为值。

### 准备阶段

这是大部分设置工作所在阶段。所有模块在被加载后都有机会准备自己。

由于来自 JSON 编码的所有属性都已经被解码，这里只需要进行额外的设置。准备阶段最常见的任务是设置访客模块。换句话说，准备宿主模块也会导致准备其访客模块，层层向下。

您可以通过 [在我们的文档中遍历 Caddy 的 JSON 结构](/docs/json/) 来了解这一点。在任何您看到 `{•••}` 的地方，都是可以使用访客模块的位置；当您点击进入其中一个时，您可以继续探索直到没有更多访客模块。

其他常见的准备任务是设置模块生命周期期间将使用的内部值，或标准化输入。例如，[`http.matchers.remote_ip`](/docs/modules/http.matchers.remote_ip) 模块在准备阶段解析从 JSON 接收的字符串输入中的 CIDR 值。这样，它就不必在每次 HTTP 请求时执行此操作，因此更高效。

验证也可以在准备阶段进行。如果模块的结果配置无效，可以在此处返回错误，从而中止整个配置加载过程。

### 使用阶段

一旦访客模块被准备和验证，它就可以被其宿主模块使用。这具体意味着什么取决于每个宿主模块。

每个模块都有一个 ID，由命名空间和该命名空间中的名称组成。例如，[`http.handlers.reverse_proxy`](/docs/modules/http.handlers.reverse_proxy) 是 HTTP 处理器，因为它位于 `http.handlers` 命名空间中，其名称是 `reverse_proxy`。`http.handlers` 命名空间中的所有模块满足相同接口，宿主模块知道这个接口。因此，`http` 应用知道如何加载和使用这些类型的模块。

### 清理阶段

当需要停止配置时，所有模块都会被卸载。如果模块分配了任何应该释放的资源，它有机会在清理阶段释放这些资源。


## 插件接入

模块——或任何 Caddy 插件——通过添加模块包的 `import` 被"接入"到 Caddy 中。通过导入包，[模块向 Caddy 核心注册自己](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc#RegisterModule)，因此当 Caddy 进程启动时，它通过名称知道每个模块。它甚至可以在模块值与名称之间建立关联，反之亦然。


<aside class="tip">

可以不修改 Caddy 代码库而添加插件。在 [readme](https://github.com/caddyserver/caddy/#with-version-information-andor-plugins) 中有说明！

</aside>


## 管理配置

更改运行中服务器的活动配置（常称为"重载"）可能很棘手，因为服务器需要高水平的并发性和数千个参数。Caddy 通过使用具有众多优点的设计优雅地解决了这个问题：

- 不中断运行中的服务
- 支持细粒度配置更改
- 只需要一个锁（在后台）
- 所有重载都是原子性、一致性、隔离性和大部分持久性的（"ACID"）
- 最小全局状态

您可以 [在这里观看关于 Caddy 2 设计原理的视频](https://www.youtube.com/watch?v=EhJO8giOqQs)。

配置重载通过准备新模块来工作，如果所有模块都成功，旧模块就会被清理。在短暂的时间内，两个配置同时运行。

每个配置都与 [上下文](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc#Context) 关联，该上下文保存所有模块状态，因此大多数状态永远不会超出配置的作用域。这对于正确性、性能和简洁性都是好消息！

然而，有时确实需要全局状态。例如，反向代理可能会跟踪其上游的健康状况；由于全局只有一个每个上游，如果每次进行小配置更改时就忘记它们，那将是不好的。幸运的是，Caddy [提供类似于语言运行时垃圾回收器的机制](https://pkg.go.dev/github.com/caddyserver/caddy/v2?tab=doc#UsagePool) 来保持全局状态整洁。

在线配置更新的显而易见方法是同步访问每个配置参数，即使在热路径中也如此。这在性能和复杂度方面极其糟糕——尤其是在大规模时——因此 Caddy 不使用这种方法。

相反，配置被视为不可变、原子性单元：要么整体替换，要么不更改任何东西。[管理 API 端点](/docs/api)—可通过遍历结构实现细粒度更改—仅修改内存中表示配置的表示，从中生成并加载全新的配置文档。这种方法在简洁性、性能和一致性方面具有巨大优势。由于只有一个锁，Caddy 可以轻松处理快速重载。
