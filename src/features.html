概述

Caddy 本质上是一个配置管理系统，可以运行各种应用程序，如 HTTP 服务器、TLS 证书管理器、PKI 工具等。它可以通过称为配置模块的插件进行扩展。

Caddy 拥有灵活强大的 HTTP 反向代理、在线配置 API 以及健壮、生产就绪的静态文件服务器，并默认通过 HTTPS 为所有站点提供服务，使用自动化的 TLS 证书。

整体程序技术规范

语言

语言选择对于 Web 服务器至关重要。语言影响开发速度和简易性、性能、测试、部署复杂度、生态系统可靠性、依赖关系、工具、错误处理和可靠性等许多方面。Go 在所有这些领域都提供了显著优势，支持快速开发、稳健的生产性能和高度可扩展性。
Go

内存安全保证

大多数服务器（NGINX、Apache、HAProxy 等）及其依赖项都是用 C 语言编写的，容易受到致命内存安全错误（如缓冲区溢出）的影响，例如 Heartbleed。像 Caddy 这样的 Go 程序可以免受整个一类安全漏洞的影响。
强

构建产物

Caddy 直接编译为原生 CPU 指令。不需要解释器；许多指令经过架构优化。静态二进制文件也更安全，因为没有动态链接。
平台原生静态二进制文件

运行时依赖

Caddy 是静态编译的。动态链接的应用程序在生产环境中容易崩溃，并且可能安全性较低，因为共享可执行资源从系统的各个位置加载。通常，Caddy 二进制文件不一定需要外部库 — 甚至不需要 libc。
无

编译时间

在消费级硬件上，标准 Caddy 构建只需几秒钟即可编译完成。这对于快速迭代、插件开发和低成本部署至关重要。其他用 C/C++ 编写的服务器可能需要几分钟。
~30 秒冷编译，
~2 秒热编译

自动 HTTPS

我们的旗舰功能，由 CertMagic 驱动。Caddy 是第一个也是唯一一个默认启用 HTTPS 的主要服务器，并自动获取和续订所有站点的证书。

完全原生、集成的自动 HTTPS 远胜于任何需要外部工具或 cron 任务的解决方案。Caddy 的证书维护是业界最好的，因为它比其他任何解决方案都更健壮、可靠和可扩展。Caddy 简化了你的基础设施而不是使其复杂化。

Sure, you can try deploying 100,000 sites with Certbot and a cron job—but if that doesn't fall over by itself, the web server will. Only Caddy is designed to massively scale TLS certificates both horizontally and vertically.

再也不用手动生成 CSR。再也不用点击电子邮件中的链接下载证书。再也不用（错误）配置 Web 服务器来使用它们。再也不用错过证书续订提醒，每隔几个月逐个续订过期证书。你甚至不用考虑证书或 TLS。

这真是自动化魔法。

OCSP

OCSP 指示证书何时被吊销。服务器应将 OCSP 响应 staple 到证书上，为客户端提供更好的安全性和隐私性。Caddy 是第一个也是唯一一个默认自动执行此操作的服务器。它还缓存响应以应对 OCSP 响应器故障，并在集群中共享。如果需要，所有这些都可以通过禁用。
带缓存的自动 OCSP staple

必须 staple

如果 CA 支持，Caddy 可以获取强制 OCSP staple 的证书。这在吊销情况下可能提供更高程度的安全性。

吊销处理

吊销的证书会自动替换。因为 Caddy staple 并刷新 OCSP 响应，它可以检测你的证书是否被吊销，如果是，它将用新证书替换它。

会话票证加固

如果攻击者窃取加密会话票证的密钥，TLS 连接就毫无意义。Caddy 已被 学术引用为唯一定期轮换这些密钥以限制攻击窗口的服务器。

证书生命周期

大多数 ACME 客户端假设为 90 天证书，或不期望短于 7 天的证书。Caddy 可以成功管理生命周期为数小时和分钟的证书。

Caddy 不是在续订前硬编码某个年龄，而是计算相对于每个证书生命周期的年龄，称为续订窗口比率。默认情况下，Caddy 在证书可用生命周期的 2/3 后续订。此比率适用于大多数有效期，但可调整。

任何生命周期

智能错误处理

如果 Caddy 无法获取证书，错误将被记录，Caddy 将指数退避并按需重试直到成功（通常最多 30 天，但可能更长）。Caddy 尽一切合理努力保持证书续订。

反向代理

Caddy 拥有世界上最灵活的多功能反向代理，具有先进的请求和响应处理、动态路由、健康检查、负载均衡、熔断等。

Caddy 代理的独特之处在于其设计。代理仅面向客户端的一侧需要是 HTTP；与后端往返的基础传输可以通过任何协议完成！

此外，我们的代理可以通过高度动态的上游进行编程。也就是说，可用的上游可以在进行中的请求期间更改！如果没有可用的后端，Caddy 可以保留请求直到有一个可用。
