概览

Caddy 本质上是一个配置管理系统，可以运行各种应用，如 HTTP 服务器、TLS 证书管理器、PKI 设施等。它可以通过称为配置模块的插件进行扩展。

Caddy 拥有一个灵活而强大的 HTTP 反向代理、在线配置 API，以及一个健壮的、可直接用于生产的静态文件服务器，默认通过 HTTPS 为所有站点提供服务，并自动管理 TLS 证书。

整体程序技术规格

编程语言

对于 Web 服务器而言，编程语言的选择至关重要。语言影响开发速度和易用性、性能、测试、部署复杂度、生态系统的可靠性、依赖关系、工具链、错误处理和可靠性等方面。Go 在这些领域都提供了显著优势，支持快速开发、健壮的生产性能和高可扩展性。
Go

内存安全保障

大多数服务器（NGINX、Apache、HAProxy 等）及其依赖项都用 C 编写，容易受到灾难性的内存安全漏洞（如缓冲区溢出）的影响，例如心脏滴血漏洞。像 Caddy 这样的 Go 程序对一整类安全漏洞具有免疫性。
强

构建产物

Caddy 直接编译为本机 CPU 指令。无需解释器；许多指令针对架构进行了优化。静态二进制文件也更安全，因为不存在动态链接。
平台原生静态二进制文件

运行时依赖

Caddy 是静态编译的。动态链接的应用程序在生产环境中容易损坏，并且可能安全性较低，因为共享的可执行资源从系统各处加载。通常，Caddy 二进制文件不需要外部库——甚至不需要 libc。
无

编译时间

在消费级硬件上，标准 Caddy 构建只需几秒钟即可完成编译。这对于快速迭代、插件开发和低成本部署至关重要。其他用 C/C++ 编写的服务器可能需要几分钟。

热构建约 2 秒

自动 HTTPS

我们的旗舰功能，由CertMagic驱动。Caddy 是第一个也是唯一一个默认启用 HTTPS 的主要服务器，并自动为所有站点获取和续期证书。

完全原生、集成的自动 HTTPS 远超任何需要外部工具或 cron 作业的方案。Caddy 的证书维护是业界最好的，因为它比其他任何解决方案都更健壮、可靠和可扩展。Caddy 简化了您的基础设施，而不是使其复杂化。

当然，您可以尝试用 Certbot 和 cron 作业部署 100,000 个站点——但如果它自己没有崩溃，Web 服务器会。只有 Caddy 被设计为可以水平和垂直大规模扩展 TLS 证书。

再也不用手动生成 CSR。再也不用点击电子邮件中的链接来下载证书。再也不用（错误）配置 Web 服务器来使用它们。再也不用在证书每几个月到期前忘记逐个续期证书。您甚至不需要考虑证书或 TLS。

这确实是自动化的魔法。

合规性

Caddy 的 TLS默认值无需额外配置就是安全的，并通过了各个行业的合规性测试。

OCSP

OCSP 指示证书何时被吊销。服务器应将 OCSP 响应 Staple 到证书中，以向客户端提供更好的安全和隐私。Caddy 是第一个也是唯一一个自动默认执行此操作的服务器。它还缓存响应以应对 OCSP 响应器故障，并在其集群中共享这些响应。如有需要，可以全部禁用。
带缓存的自动 OCSP Staple

必须 Staple

如果 CA 支持，Caddy 可以获取强制 OCSP Staple 的证书。这可能在吊销情况下提供更高程度的安全性。

吊销处理

吊销的证书会自动替换。由于 Caddy Staple 并刷新 OCSP 响应，它可以检测您的证书是否已被吊销，如果是，它将使用新证书替换它。

会话票据加固

如果攻击者窃取了加密会话票据的密钥，TLS 连接就毫无意义。Caddy 已被学术引用为唯一定期轮换这些密钥以限制攻击窗口的服务器。

证书有效期

大多数 ACME 客户端假设 90 天证书，或不期望短于 7 天的证书。Caddy 可以成功管理寿命为数小时和数分钟级别的证书。

Caddy 不是硬编码在续期前的特定年龄，而是计算相对于每个证书寿命的年龄，称为续期窗口比率。默认情况下，Caddy 在证书可用寿命的 2/3 后续期证书。此比率适用于大多数有效期，但可以调整。

任何有效期

智能错误处理

如果 Caddy 无法获取证书，将记录错误，Caddy 将指数退避并重试，直到成功（通常最多 30 天，但可能更长）。Caddy 会尽一切合理努力保持您的证书续期。

HTTP handlers

处理器是模块，可以组合在一起以按您想要的方式处理传入请求。处理器模块（与 Caddy 其余部分一样）可扩展且可插拔。我们不可能在这里列出所有处理器。

实际上，处理器与匹配器配对，匹配器根据路径、头部、查询字符串、方法等各种属性过滤或分类请求。这使您可以有选择地将任何或所有这些处理器应用于特定请求。

Reverse proxy

Caddy 拥有世界上最灵活的通用反向代理，具有先进的请求和响应处理、动态路由、健康检查、负载均衡、熔断等功能。

Caddy 代理的独特之处在于其设计。代理的客户端-facing 侧只需是 HTTP；与后端往返的底层传输可以通过任何协议实现！

此外，我们的代理可以使用高度动态的上游进行编程。也就是说，可用的上游可以在请求进行中时改变！如果没有可用后端，Caddy 可以保持请求直到有可用后端。
